# ZAP脆弱性診断スキル

## 概要

OWASP ZAP REST APIを使ってWebアプリケーションの脆弱性診断を実施するスキル。
ZAPの自動スキャン機能を最大限活用した上で、その結果をもとにClaudeが文脈を読んだ追加検証を行う。

## 前提条件

- ZAP BASE URL: `http://zap:8080`
- ZAP APIキー: 環境変数 `$ZAP_API_KEY`
- 診断対象: 環境変数 `$TARGET_URL`
- コンテキストファイル: `/home/zap/.ZAP/scan-context.context`（マウント済み）

## ZAP API 基本操作

### 疎通確認

```bash
curl -s "http://zap:8080/JSON/core/view/version/?apikey=$ZAP_API_KEY"
```

### コンテキストのインポートと確認

```bash
# Step 1: 既存コンテキスト一覧を確認
curl -s "http://zap:8080/JSON/context/view/contextList/?apikey=$ZAP_API_KEY"

# Step 2: 「zap-scan」が未ロードの場合のみインポート（重複するとbad_external_dataエラー）
curl -s "http://zap:8080/JSON/context/action/importContext/?apikey=$ZAP_API_KEY" \
  --data-urlencode "contextFile=/home/zap/.ZAP/scan-context.context"

# Step 3: コンテキストIDを取得
curl -s "http://zap:8080/JSON/context/view/context/?contextName=zap-scan&apikey=$ZAP_API_KEY"

# Step 4: 登録済みユーザーの確認
curl -s "http://zap:8080/JSON/users/view/usersList/?contextId=<contextId>&apikey=$ZAP_API_KEY"
```

### 認証確認

```bash
# 認証メソッドのロード状態を確認
curl -s "http://zap:8080/JSON/authentication/view/getAuthenticationMethod/?contextId=<contextId>&apikey=$ZAP_API_KEY"

# ユーザーのcredentials確認
curl -s "http://zap:8080/JSON/users/view/getUserById/?contextId=<contextId>&userId=<userId>&apikey=$ZAP_API_KEY"

# NOTE: 実際のログイン動作確認はPhase 1のSpider起動時にZAPが自動実行する
```

### Spider（認証済みクロール）

```bash
curl -s "http://zap:8080/JSON/spider/action/scan/?\
url=$TARGET_URL&recurse=true&contextName=zap-scan&userId=<userId>&apikey=$ZAP_API_KEY"

# 進捗確認（100になるまでポーリング）
curl -s "http://zap:8080/JSON/spider/view/status/?scanId=<scanId>&apikey=$ZAP_API_KEY"

# 発見URL一覧
curl -s "http://zap:8080/JSON/spider/view/results/?scanId=<scanId>&apikey=$ZAP_API_KEY"
```

### Active Scan（認証済み）

```bash
curl -s "http://zap:8080/JSON/ascan/action/scan/?\
url=$TARGET_URL&recurse=true&inScopeOnly=true&contextId=<contextId>&userId=<userId>&apikey=$ZAP_API_KEY"

# 進捗確認
curl -s "http://zap:8080/JSON/ascan/view/status/?scanId=<scanId>&apikey=$ZAP_API_KEY"
```

### アラート取得

```bash
curl -s "http://zap:8080/JSON/core/view/alerts/?baseurl=$TARGET_URL&apikey=$ZAP_API_KEY"
curl -s "http://zap:8080/JSON/alert/view/alertsSummary/?baseurl=$TARGET_URL&apikey=$ZAP_API_KEY"
```

### 任意リクエスト送信（追加検証用）

```bash
# GETリクエスト
curl -s "http://zap:8080/JSON/core/action/accessUrl/?\
url=<検証URL>&followRedirects=true&apikey=$ZAP_API_KEY"

# カスタムリクエスト送信
curl -s -X POST "http://zap:8080/JSON/core/action/sendRequest/?apikey=$ZAP_API_KEY" \
  --data-urlencode "request=POST <path> HTTP/1.1\r\nHost: <host>\r\nContent-Type: application/json\r\n\r\n<body>"
```

### 送受信履歴の確認

```bash
curl -s "http://zap:8080/JSON/core/view/messages/?baseurl=$TARGET_URL&apikey=$ZAP_API_KEY"
curl -s "http://zap:8080/JSON/core/view/message/?id=<messageId>&apikey=$ZAP_API_KEY"
```

## 診断フロー

### Phase 0: 起動確認

1. ZAP疎通確認
2. 既存コンテキスト一覧を確認
3. `zap-scan` が未ロードの場合のみ `scan-context.context` をインポート
4. コンテキストIDとユーザーIDを取得
5. 認証メソッドとcredentialsのロード状態を確認

```
⚠️ 以下の場合は診断を中止して報告:
- importContextがbad_external_data以外のエラーで失敗
- ユーザーが0件
- 認証メソッドがNOT_SET状態

→「zap-config/scan-context.contextが正しくロードされていません。
  ホスト側でZAP GUIを使ってコンテキストを再エクスポートしてください。」
```

### Phase 1: 初期調査

1. contextId・userId を指定して認証済みSpiderを実行
2. 発見URLを確認して以下の観点で分類する

```
確認ポイント:
- 認証後にしか見えないエンドポイントが取得できているか
- APIエンドポイントはあるか（/api/, /rest/ など）
- パスに数値IDを含むエンドポイントはあるか（BOLA候補）
- 価格・数量・残高・クーポンを扱うエンドポイントはあるか
- ステップを踏む必要がある操作フロー（決済・注文・返金）はあるか
- ユーザー入力を保存して後から表示する機能はあるか（Stored XSS候補）
- ファイルアップロード機能はあるか
- 管理者向けページはあるか
- GraphQLエンドポイントはあるか（/graphql）→ Introspectionで操作一覧を取得
- パスワード変更・メールアドレス変更等の認証情報変更機能はあるか
- 認証にJWTを使用しているか（alg:none検証の対象）
```
Spider完了後、上記の分類結果をもとにPhase 3-A・3-Bの検証対象リストを作成してから次のフェーズに進む。

### Phase 2: 自動スキャン（ZAPに任せる）

1. 認証済みユーザーでActive Scanを実行
2. 完了まで進捗を定期的に報告
3. アラート一覧を取得してリスク別に集計
4. 追加検証の優先度を判断

```
優先度の判断基準:
- High   → 必ず追加検証を実施
- Medium → 内容を見て判断（False Positiveの可能性が高ければスキップ）
- Low    → レポートに記載のみ
- Info   → レポートに記載のみ
```

### Phase 3-A: 手動追加検証

**ZAPのアラート有無にかかわらず、以下の項目は必ずClaudeが能動的に確認する。**
ZAPが検出していても「再現確認とFalse Positive排除」、未検出でも「該当機能があれば検証する」。

#### A-1: Stored XSS

Spider結果からユーザー入力をサーバーに保存して後から表示する機能を特定し、スクリプトタグが格納・返却されるかを確認する。

```
対象の見つけ方:
- Spider結果から POST/PUT で文字列を受け取るエンドポイントを列挙
  （レビュー・コメント・プロフィール・問い合わせ等）
- 該当エンドポイントにペイロードを送信してGETで取得し、タグがエスケープされずに返るか確認

検証手順:
1. 対象エンドポイントに以下を送信
   {"<フィールド名>": "<script>alert('XSS')</script>"}
2. 保存後にGETで取得し、スクリプトタグがそのまま返却されるか確認
3. HTMLエスケープ（&lt;script&gt;）されていれば正常

判定基準:
- タグがそのまま返却される → 脆弱性あり（High）
- エスケープされる        → 正常
```

#### A-2: Reflected XSS / DOM XSS

ZAP検出済みアラートの再現確認に加え、以下の観点でSpider結果から未検出箇所を探す。

```
対象の見つけ方:
- クエリパラメータをレスポンスにそのまま反映するエンドポイント
- エラーページや検索結果ページで入力値が表示される箇所

検証手順:
1. パラメータに <script>alert(1)</script> や "><img src=x onerror=alert(1)> を送信
2. レスポンスボディで入力値がエスケープされずに出力されているか確認

判定基準:
- スクリプトが実行可能な形で返る → 脆弱性あり（High〜中）
- エスケープされる               → 正常
```

#### A-3: SQLインジェクション

ZAP検出済みの場合は再現確認とFalse Positive排除。未検出でも文字列検索・ログインフォーム等は必ず確認する。

```
検証手順:
1. 文字列パラメータに ' を送信してHTTP 500やSQLエラーが返るか確認（エラーベース）
2. ' OR '1'='1 で全件取得できるか確認
3. 時間ベース（SLEEP等）でブラインドSQLiを確認

判定基準:
- SQLエラーメッセージが返る         → 脆弱性あり（High）
- 条件変更で件数・内容が変わる      → 脆弱性あり（High）
- エラーなく正常応答のみ            → 要追加確認（ブラインド試行）
```

#### A-4: 認証・JWT検証

認証トークンを使用しているアプリでは以下を必ず確認する。

```
確認項目:
1. JWT alg:none — ヘッダーのalgをnoneに書き換えた無署名トークンが受け入れられるか
2. JWT HS256/RS256混在 — RS256公開鍵をHS256の秘密鍵として使う攻撃が成立するか
3. トークンペイロードへの機密情報含有 — Base64デコードでパスワードハッシュ等が含まれていないか
4. パスワード変更時の現パスワード検証 — currentパラメータ省略で変更可能かを確認
5. ログアウト後のトークン無効化 — ログアウト後のトークンでAPIアクセスが可能か

判定基準:
- alg:none が受け入れられる                 → 脆弱性あり（緊急）
- ペイロードに機密情報含有                  → 脆弱性あり（中）
- 現パスワード確認なしで変更可能            → 脆弱性あり（高）
- ログアウト後もトークンが有効              → 脆弱性あり（高）
```

#### A-5: ユーザー列挙・情報開示

認証ユーザーの範囲を超えて情報が取得できるエンドポイントを確認する。

```
確認項目:
1. /api/Users 等のユーザー一覧エンドポイントに一般ユーザーでアクセスできるか
2. エラーメッセージに内部情報（スタックトレース・DBスキーマ・ファイルパス）が含まれるか
3. レスポンスに不要なフィールド（パスワードハッシュ・内部ID・ロール情報）が含まれるか

判定基準:
- 一般ユーザーで全ユーザーデータが取得可能  → 脆弱性あり（高）
- エラーに内部情報が含まれる               → 脆弱性あり（中〜低）
```

#### A-6: セキュリティヘッダー確認

レスポンスヘッダーを一括確認する。ZAPが検出していても再確認すること。

```
確認項目:
- Content-Security-Policy（未設定は中リスク）
- X-Frame-Options（未設定はクリックジャッキングリスク）
- X-Content-Type-Options: nosniff（未設定は低リスク）
- Strict-Transport-Security（HTTP環境では低リスク）
- Set-Cookie の Secure / HttpOnly フラグ

確認コマンド:
curl -I <TARGET_URL>
```

#### A-7: エラー応答の深掘り

Phase 3-B等でHTTP 500や予期しないエラーが返った場合、「検証不可」で終わらせず原因を調査する。

```
手順:
1. エラーが返ったリクエストのレスポンスボディを確認
   - スタックトレースが含まれていれば情報開示として記録
   - SQLエラーが含まれていればSQLiの可能性として追加検証
2. パラメータの形式・型・必須項目を変えて再送し、正常応答が得られるパターンを特定
3. 正常応答が得られた上で脆弱性検証を継続する

記録方針:
- 正常応答パターンが見つかった → 検証を継続してビジネスロジック検証結果に記録
- どのパターンでもエラー → 「機能不全のため検証不可」として理由とともに記録
```

### Phase 3-B: ビジネスロジック検証

**ZAPは「リクエスト/レスポンスのパターン異常」しか検知できない。**
アプリの仕様として「本来許可されるべきでない操作」はClaudeが能動的に検証する。

Phase 1および2の結果をもとに、診断対象サイトについて「ビジネスにおいてどのような目的を達成するためにどういった用途で用いられるサイトか」を定義する。
その後、以下の観点ごとに対象エンドポイントを特定してから検証を実施すること。
対象が見つからない観点はスキップしてよい。

#### 観点1: BOLA（Broken Object Level Authorization）

パスに数値IDや識別子を含むエンドポイントを特定し、他ユーザーのリソースにアクセスできるかを確認する。

```
検証手順:
1. Spider結果から /api/<resource>/<id> 形式のエンドポイントを列挙
2. テストユーザーAで自分のリソースにアクセスして成功することを確認
3. 同じリクエストを他ユーザーのIDに書き換えて送信
4. HTTP 200 かつ他ユーザーのデータが返ればBOLA確定

判定基準:
- 200 + 他ユーザーデータ → 脆弱性あり（High）
- 403 / 401             → 正常
- 404                   → IDが存在しないか、アクセス制御で隠蔽（要追加確認）
```

#### 観点2: 数値・金額操作

```
検証手順:
1. price / quantity / amount / balance 等のパラメータを持つエンドポイントを特定
2. 負の値・ゼロ・極端に大きな値・小数・文字列・nullを順番に試す
3. レスポンスのステータスと処理結果を確認

判定基準:
- 負の数量で注文が成立 → 脆弱性あり（High）
- 負の金額で残高増加  → 脆弱性あり（Critical）
- エラーになる        → 正常
```

#### 観点3: フロー強制（ステップスキップ）

```
検証手順:
1. 多段フローを持つ機能を特定（注文・決済・パスワードリセット等）
2. 前段のステップを踏まずに後段のエンドポイントを直接呼び出す
3. 操作が成立するかを確認

判定基準:
- ステップスキップで処理が成立 → 脆弱性あり（Medium〜High）
- 前段の状態がない場合にエラー → 正常
```

#### 観点4: クーポン・割引・特典の重複適用

```
検証手順:
1. クーポン・割引・プロモーション関連のエンドポイントを特定
2. テストユーザーで一度正常適用して成功を確認
3. セッションを維持したまま同一コードを再送信
4. 別のタイミング（注文後など）でも再送信を試みる

判定基準:
- 2回目以降も適用成功   → 脆弱性あり（Medium）
- 2回目以降はエラー応答 → 正常
```

#### 観点5: 垂直権限昇格

```
検証手順:
1. /admin / /api/admin / /management 等の管理系エンドポイントを列挙
2. 一般ユーザーのJWTトークンでそのエンドポイントにアクセス
3. GETで情報取得できるか、POST/PATCHで操作できるかを確認

判定基準:
- 一般ユーザーで管理機能が操作可能 → 脆弱性あり（Critical〜High）
- 403 / 401                        → 正常
```

#### 観点6: 機能レベルのアクセス制御不備（BFLA）

```
検証手順:
1. ユーザーAでリソース（注文・レビュー・プロフィール等）を作成
2. ユーザーBのセッションに切り替えて、ユーザーAのリソースIDに対して
   PUT / PATCH / DELETE を送信
3. 操作が成立するかを確認

判定基準:
- 他ユーザーのリソースを操作できる → 脆弱性あり（High）
- 403 / 404                        → 正常
```

#### ビジネスロジック検証の記録方針

各観点について以下を記録すること：

```
- 対象エンドポイントが存在したか（なければ「対象なし・スキップ」と記録）
- 実施したリクエストの概要（メソッド・URL・パラメータ名・検査値）
- レスポンスの概要（ステータスコード・応答内容）
- 判定結果（脆弱性あり / 正常 / 要追加確認）
```

## 注意事項

- コンテキストのスコープ外URLには絶対にアクセスしない
- Active Scan中はサーバーに負荷がかかるため進捗を定期的に報告する
- 認証情報が発見された場合はレポートに [REDACTED] と記載する
- False Positiveと判断した場合は必ずその理由を記録する
- ビジネスロジック検証で対象エンドポイントが存在しない場合は「対象なし」として記録し、検証をスキップしてよい
- 総合評価は検出された最高リスクレベルに基づいて決定する（緊急が1件でもあればE）
- 指摘事項の順序は必ずリスクレベルの高い順（緊急→高→中→低）とする
- 脆弱性診断に不慣れな開発者が読んでも理解できるレベルの内容に落とし込む
